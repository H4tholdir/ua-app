# ADVERSARIAL AUDIT — UÀ PWA
**Date:** 2026-05-21  
**Reviewer stance:** Assume overconfidence. Find what was missed.

## TOP 5 MISSED RISKS
1. **Create-time cross-tenant foreign-key injection** — affected files: `src/app/api/lavori/route.ts` `POST`, `supabase/schema.sql` `CREATE TABLE lavori`. Concrete scenario: an authenticated user in Lab A sends `POST /api/lavori` with `cliente_id`, `paziente_id` or `tecnico_id` belonging to Lab B. The insert uses the service client and accepts those IDs directly at `src/app/api/lavori/route.ts:138-165`; unlike the PATCH route, there is no FK ownership validation before insert. The DB schema only declares plain FKs (`cliente_id UUID NOT NULL REFERENCES clienti(id)`, `paziente_id UUID REFERENCES pazienti(id)`, `tecnico_id UUID REFERENCES tecnici(id)`) at `supabase/schema.sql:866-871`, not composite `(id, laboratorio_id)` constraints. Severity: High. Original agents missed it because they praised the PATCH fix: CODEX-PERSPECTIVE says "Tenant isolation with `laboratorio_id` presente nella stragrande maggioranza delle query" and specifically cites PATCH FK validation, but did not apply the same adversarial check to creation.

2. **Service-role updates still depend on prior checks instead of tenant-scoped writes** — affected files: `src/app/api/lavori/[id]/segnala/route.ts`, `src/app/api/lavori/[id]/segnala/risolvi/route.ts`, `src/lib/consegna/orchestrate.ts`, `src/lib/pdf/generate-buono.ts`. Concrete scenario: because these handlers bypass RLS through `getServiceClient()`, a maintenance bug, stale object, or race after the read check can update by bare `id`. `segnala` verifies ownership at `src/app/api/lavori/[id]/segnala/route.ts:61-68` but updates only `.eq('id', id)` at `src/app/api/lavori/[id]/segnala/route.ts:74-85`; `risolvi` repeats the same pattern at `src/app/api/lavori/[id]/segnala/risolvi/route.ts:35-52`; delivery status update uses `.eq('id', lavoro_id)` without `.eq('laboratorio_id', laboratorio_id)` at `src/lib/consegna/orchestrate.ts:195-207`; `generateBuono` updates `.eq('id', lavoro.id)` at `src/lib/pdf/generate-buono.ts:48-53`. Severity: High. Original agents partially found one instance, then downplayed it as "BASSA"; the missed point is systemic: once RLS is bypassed, every write must carry the tenant predicate, not just the read before it.

3. **Delivery is not actually atomic despite the audit calling it atomic** — affected files: `src/lib/consegna/orchestrate.ts`, `src/lib/pdf/generate-ddc.ts`, `src/lib/pdf/generate-buono.ts`, `src/app/api/lavori/[id]/annulla-consegna/route.ts`. Concrete scenario: the orchestrator generates and stores a DdC at `src/lib/consegna/orchestrate.ts:155-170` and `src/lib/pdf/generate-ddc.ts:77-104`, generates and stores a buono at `src/lib/consegna/orchestrate.ts:173-188` and `src/lib/pdf/generate-buono.ts:48-53`, then only later marks the work delivered at `src/lib/consegna/orchestrate.ts:190-207`. If the final update fails, there can be legal/compliance documents for a job that is not delivered. The undo route only resets the job and marks DdC annulled at `src/app/api/lavori/[id]/annulla-consegna/route.ts:63-87`; it does not clean generated buono URLs or inventory scarichi. Severity: High. Original agents missed it because SINTESI-ORCHESTRATORE claims "flusso consegna atomico con MDR"; the counter-evidence is that the flow is a sequence of side effects, not a DB transaction or a resumable job.

4. **CSRF coverage is not "almost complete" on real mutating routes** — affected files: `src/app/api/lavori/[id]/segnala/route.ts`, `src/app/api/lavori/[id]/segnala/risolvi/route.ts`, `src/app/api/admin/labs/[id]/hard-delete/route.ts`, `src/lib/utils/csrf.ts`. Concrete scenario: a malicious same-browser page can submit state-changing requests against the user's session for problem reporting/resolution because both `segnala` and `risolvi` expose mutating handlers with no `isSameOrigin()` call at `src/app/api/lavori/[id]/segnala/route.ts:16-24` and `src/app/api/lavori/[id]/segnala/risolvi/route.ts:7-15`. The hard-delete route implements a weaker substring check, `origin.includes(host.split(':')[0])`, at `src/app/api/admin/labs/[id]/hard-delete/route.ts:9-12`, instead of the exact host comparison in `src/lib/utils/csrf.ts:7-15`. Severity: Medium. Original agents missed the practical attack path because CODEX-PERSPECTIVE frames `segnala/risolvi` as "basso" and `hard-delete` as protected by role, but CSRF defenses should not vary by perceived endpoint importance.

5. **The UI can deliver stale or unsaved MDR data, and the server will never know** — affected files: `src/components/features/lavori/LavoroFormClient.tsx`, `src/hooks/useLavoroForm.ts`, `src/components/features/lavori/ConsegnaButton.tsx`, `src/lib/consegna/precheck.ts`. Concrete scenario: the form keeps edits client-side, marks dirty at `src/hooks/useLavoroForm.ts:76-81`, and only autosaves after 30 seconds at `src/hooks/useLavoroForm.ts:83-102`. The delivery navigation button remains enabled while dirty and simply pushes to `/consegna` at `src/components/features/lavori/LavoroFormClient.tsx:345-371`, so a user can enter acceptance/MDR data and immediately deliver using stale server data. Separately, `ConsegnaButton` does not enter `loading` until after the best-effort material precheck at `src/components/features/lavori/ConsegnaButton.tsx:100-118`, leaving a double-tap window. Severity: Medium. Original agents missed it because they evaluated server precheck (`src/lib/consegna/precheck.ts:18-95`) as if it validates the user's latest edits; it only validates what was previously saved.

## 3 OVERENGINEERED AREAS
1. **Manual service-role multi-tenancy layer** — files involved: `src/app/api/lavori/route.ts`, `src/app/api/lavori/[id]/route.ts`, `src/app/api/lavori/[id]/consegna/route.ts`, most `src/app/api/**/route.ts`. It adds complexity without proportional value because every route reimplements auth lookup, lab lookup, role rules, tenant predicates, and CSRF manually; the result is predictable drift such as create-time FK validation missing in `POST /api/lavori` while present in PATCH. Simpler alternative: default to user-scoped Supabase clients/RLS for CRUD and isolate service-role use inside narrow RPCs that take `actor_user_id` and enforce tenant/role checks in one place.

2. **TypeScript delivery orchestrator as transaction coordinator** — files involved: `src/lib/consegna/orchestrate.ts`, `src/lib/pdf/generate-ddc.ts`, `src/lib/pdf/generate-buono.ts`, `src/app/api/lavori/[id]/annulla-consegna/route.ts`. It adds complexity without proportional value because it mixes lock acquisition, PDF rendering, storage upload, state transition, invoice draft creation, WhatsApp link generation, and inventory decrement in one long function, but cannot roll back the external side effects it creates. Simpler alternative: use a DB-backed delivery job/state table with explicit phases and idempotent retries, or a single RPC for the state transition plus outbox rows for PDF/inventory/invoice work.

3. **Nine-tab client form with autosave illusion** — files involved: `src/components/features/lavori/form/LavoroFormShell.tsx`, `src/components/features/lavori/LavoroFormClient.tsx`, `src/hooks/useLavoroForm.ts`. It adds complexity without proportional value because users see a persistent "CONSEGNA" action while the actual saved state can lag 30 seconds, and the hook saves the whole `data` object rather than a minimal validated diff. Simpler alternative: explicit save-before-deliver flow, route-level dirty blocker, and per-section submit handlers with focused validation.

## HONEST OVERALL VERDICT
This codebase is not production-ready for a multi-tenant SaaS, even if it may be usable for a friendly first lab. The single biggest blocker is the service-role architecture: too many handlers bypass RLS and rely on hand-written tenant checks that are inconsistently applied. The confirmed create-time FK gap in `POST /api/lavori` is enough to reject the "RLS solido" conclusion.

The competent attacker would first look for ID-based tenant confusion, not for visual PWA bugs. They would enumerate or obtain UUIDs through normal workflow leaks, then try cross-tenant `cliente_id`, `paziente_id`, `tecnico_id`, and state-changing service-role endpoints where the final write lacks `laboratorio_id`.

The delivery flow is also over-trusted. It is marketed in the audit as atomic, but the code performs irreversible side effects before the final state transition and runs invoice/material updates outside the core success path. That is a maintenance trap: the first intermittent storage, PDF, or DB failure will create contradictory legal, inventory, and workflow state.
